Linux-历史发展历程

# Unix 历史

Unix 发展史

  • 早期计算机的输入设备只有卡片阅读机，输出设备只有打印机，用户也无法与操作系统互动（属于批次型操作系统）。那个时候写程序必须要将程序相关的信息在读卡纸上面打洞，然后再将读卡纸插入卡片阅读机来将信息读入主机中运算。如果程序有个地方写错，重新读卡就很麻烦。加上主机少，用户众多，光是等待，就耗去很多时间
  • 在那之后，由于硬件和操作系统的改良，使得后来可以使用键盘进行信息的输入。在 1960 年代初麻省理工学院（MIT）发展了所谓的『兼容分时系统（CTSS:Compatible Time-Sharing System）』，它可以让大型主机通过提供数个终端机（Terminal）以联机进行主机，来利用主机的资源进行运算工作。即多个使用者在某一段时间内分别使用 CPU 的资源，感觉上大家是同时使用该主机的资源，事实上是 CPU 在每个使用者的工作之间进行切换。注意，此时终端机只具有输入输出的功能，本身完全不具任何运算或者软件安装的能力，而且比较先进的主机大概也只能提供 30 个不到的终端机而已
  • 为了加强大型主机的功能，以让主机的资源可以桶更多使用者来利用，所以在 1965 年前后，由贝尔实验室（Bell）、麻省理工学院及奇异公司（GE，或称通用电器）共同发起了 Multics计划。Multics 计划的目的是想要让大型主机可以达成提供 300 个以上的终端机联机使用的目标，不过到了 1969 年前后，计划进度落后，资金也短缺，所以该计划虽然继续在研究，但贝尔实验室还是退出了该计划的研究工作
  • Ken Thompson 是原本参加 Multics 计划的其中一位，他因自己的需要，希望开发一个小小的操作系统以提供自己的需求。经过四星期的奋斗，他终以汇编语言（Assembler）写出了一组核心程序，同时包括一些核心工具程序，一级一个小小的文件系统，那个系统就是 Unix 的原型。当时 Thompson 将 Multics 庞大的复杂系统简化了不少，于是同实验室的朋友都戏称这个系统为 Unics。Thompson 的这个文件系统有两个重要大概念，这些概念在后来对 Linux 的发展有相当重要的影响:① 所有的程序或系统装置都是文件 ② 不管建构编辑器还是附属文件，所写的程序只有一个目的，且要有效的完成目标
  • 由于 Thompson 写的那个操作系统实在太好用了，所以在贝尔实验室内部广为流传，并且数度经过改版。但因为 Unics 本来是以汇编语言写成的，所以每次要安装不同的机器都得要重新编写汇编语言，真不方便（汇编语言具有专一性，加上当时的计算机机器架构都不太相同）。后来 Thompson 和 Ritchie 合作想将 Unics 改以高阶程序语言来编写。当时现成的高阶程序语言有 B 语言，但是由 B 语言所编译出来的核心效能不是很好。后来 Dennis Ritchine 将 B 语言重新改写成为 C 语言，再以 C 语言重新改写与编译 Unics 的核心，最后于 1973 年正名与发行出 Unix 的正式版本。需要特别强调的是，由于 Unix 是用 C 语言写的，相对于汇编语言需要与硬件有密切的配合，高阶 C 语言与硬件的相关性就没有那么大了。这个改变也使得 Unix 很容易被移植到不同的机器上面
  • 贝尔实验室隶属于美国电信大厂 AT&T，它对 Unix 是采取较开放的态度，此外 Unix 是以 C 语言编写的，理论上是具有可移植性的。亦即只要取得 Unix 的原始码（Source Code），并且针对大型主机的特性加以修订原有的原始码，就可能将 Unix 移植到另一部不同的主机上头了。所以在 1973 年以后，Unix 便得以与学术界合作开发，最重要的接触就是与加州伯克莱（Berkeley）大学的合作了，伯克莱大学的 Bill Joy 在取得 Unix 的核心原始码后，着手修改成何时自己机器的版本，并且同时增加了很多工具软件与编译程序，最终于 1977 年，将它命名为 BSD:Berkley Software Distribution 并发布。这个 BSD 是 Unix 很重要的一个分支，他本人也是 SUN 公司的创办人，SUN 公司就是以 BSD 发展的核心进行自己的商业 Unix 版本的发展的
  • 操作系统的核心（Kernel）必须要跟硬件配合，以提供及控制硬件的资源进行良好的工作。而在早期每一家生产计算机硬件的公司没有所谓的『协议』概念，所以每一个计算机公司出产的硬件各不相同。它们需要为自己的计算机硬件开发合适的 Unix 系统，但每一家公司自己出的 Unix 在架构上面大同小异，并且仅能支持自身公司的硬件，所以早先的 Unix 只能与服务器或者是大型工作站划上等号（由于没有厂商针对个人计算机设计 Unix 系统，因此在早期并没有支持个人计算机的 Unix 操作系统的出现;Unix 强调的是多人多任务的环境，而早期 286 个人计算机架构下的 CPU 是没有能力达到多任务的作业，因此并没有人对移植 Unix 到 x86 的计算机上感兴趣）。但到了 1979 年，AT&T 推出 SystemV 第七版 Unix，这一版支持 x86 架构的个人计算机系统。不过 AT&T 由于商业的考虑，想将 Unix 版权收回去，因此提出了『不可对学生提供原始码』的严格限制
  • 关于 1979 年的版权声明中，影响最大的当然就是学校教 Unix 核心原始码相关学问的教授了，这个问题对于 Andrew Tanenbaum 教授来说，是在很伤脑筋。于是他自己在 1984 年到 1986 年动手编写并完成了 Minix 的核心程序，并于次年出版 Minix 相关书籍（Minix 就是 Mini 的 Unix 系统）。在撰写过程中，为了避免版权纠纷，他完全不看 Unix 的核心原始码，并且强调 Minix 必须能够与 Unix 兼容。这个 Minix 版本并不是完全免费，无法在网络上提供下载，必须通过磁盘/磁带购买，磁盘会附上 Minix 的原始码，这意味着使用者可以学习 Minix 的核心程序设计概念。Minix 没有在网络上流传，所以 Minix 的传播速度并不是很快。此外，Minix 的开发者认为 Minix 主要用在教育用途上面，所以对于 Minix 是点到为止，使用者的要求/需求无法得到满足
  • Rickard Mathew Stallman（史托曼）在 1984 年发起 GNU 计划。1983 年以后，他发现自己过去使用的 Lisp 操作系统是麻省理工学院的专利软件，是无法共享的，于是他便放弃了 Lisp 这个系统。后来，他接触到了 Unix 这个系统，并且发现 Unix 在理论与实际上都可以在不同的机器间进行移植。虽然 Unix 依旧是专利软件，但至少 Unix 架构上还是比较开放的，于是他开始转而使用 Unix 系统
    • 1984 年史托曼开始 GNU 计划，这个计划目的是建立一个自由、开放的 Unix 操作系统（Free Unix）。但建立一个操作系统谈何容易，他决定反其道而行『先写可以在 Unix 上面运行的小程序』，于是史托曼开始参考 Unix 上面现有的软件，并依据这些软件的作用开发出功能相同的软件，且开发期间绝不看其他软件的原始码，以避免吃官司。后来一堆人知道免费的 GNU 软件，并且实际使用后发现与原有的专利软件也差不太多，于是便转而使用 GNU 软件，于是 GNU 计划主键打开知名度
    • 虽然 GNU 计划逐渐打开知名度，但能见度还是不够。这是史托曼又想『不论是什么软件，都得要进行编译成为二进制文件后才能执行，如果能够写出一个不错的编译程序，那不就是大家想要的软件了吗』。因此他开始编写 C 语言的编译程序，那就是现在相当有名的 GCC （GNU C Comiler）。当时 C 语言编译程序版本众多，但都是专利软件，如果他写的 C 编译程序够棒，那将会大大的让 GNU 计划出现在众人眼前!但开始编写时并不顺利，为此他先转而将他原来就已经写过的 Emacs编辑器 写成可以在 Unix 上面跑的软件，并公布原始码。Emacs 是一种编辑器，它可以在用户编写程序的过程中就进行程序语法的检验，此功能能减少程序员排错的时间。此时因特网尚未流行，所以是托曼接 Emacs 以磁带出售赚了一点钱，进而开始权力编写其他软件。并成立了 自由软件基金会（FSF:Free Software Foundation），请更多工程师一起编写软件。最终完成了 GCC，这比 Emacs 还更有帮助。此外，他还编写了可以被使用来操作操作系统的基本接口 BASH Shell等。这造就后来很多的软件开发者可以藉由这些基础的工具来进行程序开发，进一步壮大了自由软件团体。到了 1985 年，为了避免 GNU 所开发的自由软件被其他人所利用而成为专利软件，他和律师草拟了有名的 通用公共许可证（GPL:General Public License）
    • 不过，对 GNU 的最初构想『建立一个自由的 Unix 操作系统』来说，仅靠这些程序仍无法满足，因为当下没有『自由的 Unix 核心』，所以这些软件仍只能在那些有专利的 Unix 平台上工作，一直到 Linux 的出现
  • 有鉴于图形用户接口（GUI:Graphical User Interface）的需求日益加重，在 1984 年由 MIT 与其他第三方首次发表了 X Window System，并且在 1988 年成立了非盈利性质的 XFree86 这个组织。而这个 XFree86 的 GUI 界面更在后来被整合于 Linux 的核心 1.0 中

# Linux 历史

Linux 发展史

  • 芬兰人 Linus Torvalds 的外祖父是赫尔辛基大学的统计学家，他的外祖父为了让自己的小孙子能够学点东西，所以从小就将托瓦兹带到身边来管理一些微计算机。在这个时期，托瓦兹接触了汇编语言（那是一种直接与芯片沟通的程序语言，必须要很了解硬件的架构，否则很难以汇编语言编写程序的）
  • 在 1988 年间，托瓦兹顺利进入了赫尔辛基大学，并选读了计算机科学系。在就学期间，因为学业的需要和自己的兴趣，托瓦兹接触到了 Unix 这个操作系统。当时整个赫尔辛基只有一部最新的 Unix 系统，同时仅提供 16 个终端机。在这种情况下，实在很难满足托瓦兹的需求，因为光是等待使用 Unix 就很耗时。为此他不禁想到『为何我不自己搞一部 Unix 来玩?』
  • 不久之后，他就知道有一个类似 Unix 的系统，并且与 Unix 完全兼容，还可以在 Intel386 机器上面跑的操作系统——Minix。他在购买了最新的 Intel386 的个人计算机后，就立即安装了 Minix 这个操作系统。托瓦兹也通过 Minix 的原始码学习到了很多的核心程序设计的设计概念（早期 Intel x86 架构计算机不是很受重视的原因是 x86 的芯片对于多任务的处理不佳，CPU 在不同的工作之间切换不是很顺畅。但这个情况在 386 计算机推出后，有很大的改善）
  • 探索完 386 硬件性能之后，终于拿到 Minix 并且安装在托瓦兹的 386 计算机之上后，托瓦兹发现 Minix 虽然真的很棒，但谭宁邦教授就是不愿意进行功能的加强，导致一堆工程师在操作系统上面欲求不满。这个时候托瓦兹就想『既然如此，那我何不自己来改写一个自己想要的操作系统』，于是他就开始了核心程序的编写。编写程序需要什么呢?首先需要的是能够进行工作的环境，再来则是可以将原始码编译成为可执行文件的编译程序。好在 GNU 计划提供的 bash 工作环境软件以及 GCC 编译程序等自由软件，让托瓦兹得以顺利的编写核心程序。他参考 Minix 的设计理念与书上的程序代码，然后仔细研究出 386 个人计算机的效能优化，使用 GNU 的自由软件将核心程序代码与 386 紧紧的结合在一起，最终写出他所需要的核心程序。这个程序不仅可以在 386 上面顺利跑起来，还可以读取 Minix 的文件系统
  • 不过还不够，他希望这个程序可以获得大家的一些修改建议，于是他便将这个核心放置在网络上提供大家下载。因为托瓦兹放置核心的 FTP 网站的目录为 Linux，所以大家便称这个核心为 Linux（Linux 第一个和新版本为 0.02）。同时，为了让自己的 Linux 能够兼容于 Unix 系统，于是托瓦兹开始将一些能够在 Unix 上面运作的软件拿来在 Linux 上面跑。不过他发现很多的软件无法在 linux 这个核心上面运作。这个时候它有两种做法:一种是修改软件，让该软件可以在 Linux 上面跑;另一种则是修改 Linux，让 Linux 符合软件能够运作的规范。由于 Linux 希望能够兼容于 Unix，于是托瓦兹选择了第二个做法『修改 Linux』，为了让所有的软件都可以在 Linux 上执行，于是托瓦兹开始参考标准的 POSIX规范（POSIX 重点在于规范核心与应用程序之间的接口，这是 IEEE 所发布的一项标准），因为 POSIX 标准主要是针对 Unix 与一些软件运行时候的标准规范，只要依据这些标准规范来设计的核心与软件，理论上就可以搭配在一起执行了。而 Linux 的发展就是依据这个 POSIX 的标准规范，Unix 上面的软件也是遵循这个规范来设计的。如此一来，让 Linux 很容易就与 Unix 兼容共享互有的软件了。同时，因为 Linux 直接放置在网络上提供大家下载，所以在传播的速度上非常快，导致 Linux 的使用率大增
  • Linux 能够成功除了托瓦兹个人的理念与力量之外，其实还有个最重要的团队
    • 单一个人维护阶段: Linux 虽然是托瓦兹发明的，但如果单靠托瓦兹自己一个人的话，那么 Linux 要茁壮成长也很困难。好在托瓦兹将推出的 Linux 核心放置在 FTP 上面，并请告知大家新的版本信息，等到用户下载了这个核心并且安装之后，如果发生问题，使用者可以主动联系给托瓦兹。在托瓦兹能够解决的问题范围内，他都能很快速的进行 Linux 核心的更新与排错
    • 广大人员加入阶段: 不过托瓦兹总是有些硬件无法取得，那么他当然无法帮助进行驱动程序的编写与相关软件的改良。这个时候其他能够解决问题的人员便跳出来说『这个硬件我有，我来帮忙写相关的驱动程序』。因为 Linux 的核心是开源的，人们可以很容易就能够跟随 Linux 的原来设计架构，并且写出兼容的驱动程序或者软件，并且加以测试。只要测试可以运行，并且没有什么主要的大问题，那么托瓦兹就会很乐意的将别人写的程序代码加入核心中。另外，为因应这种随时都有程序代码加入的状况，于是 Linux 便逐渐发展成具有模块的功能。大大增加了 Linux 核心的可维护能力
    • 核心功能细部分工发展阶段: 后来因为 Linux 核心加入了太多的功能，光靠托瓦兹一个人进行核心的实际测试并加入核心原始程序实在太费力。结果就是有很多朋友跳出来帮忙这个前置工作。这些重要的扶手会先将其他人的修补程序或者新功能的程序代码进行测试，并且结果上传给托瓦兹看，让托瓦兹做最后核心加入的原始码的选择与整并。这个分层负责的结果，让 Linux 的发展更加的容易
  • 在 1994 年，终于完成了 Linux 的核心正式版 version1.0。这一版还同时加入了 X Window System 的支持。于 1996 年完成了 2.0 版本，2011 年放出 3.0 版本，2015 年放出 4.0 版本，发展相当迅速。此外，托瓦兹指明了企鹅作为 Linux 的吉祥物

# Linux 的发布商版本

Linux 核心版本与 Linux 发布商版本（distribution）并不相同

  • 虽然托瓦兹负责开发的 Linux 仅具有 Kernel 和 Kernel 提供的工具，不过很多软件已经可以在 Linux 上面运行了。因此『Linux+各种软件』就可以完成一个相当完整的操作系统了。不过因为 Linux 早期都是有黑客工程师所开发维护的，他们并没有考虑到一般使用者的能力（将原始码安装到 x86 计算机上面成为可执行的二进制文件，这个过程可不是人人都会的）
  • 为了让使用者能够接触到 Linux，于是很多的商业公司或非盈利团体，就将 Linux Kernel（含 Tools）与可运行的软件整合起来，加上自己具有创意的工具程序，这个工具程序可以让用户以光盘/DVD 的形式直接安装管理 Linux 系统。这个『Kernel + Softwares + Tools + 可完全安装程序』称之为 Linux distribution，中文叫可完全安装套件，或 Linux 发布商套件等
  • 不过由于发展 Linux distributions 的公司实在太多了。为此为了让所有的 Linux distributions 开发不至于差异太大，且让这些开发商在开发的时候有所依据，还有 Linux Standard Base（LSB）等标准来规范开发者，以及目录架构的 File System Hierarchy Standard（FHS）标准规范。唯一差别的，可能就是该开发者自家所开发出来的管理工具，以及套件管理的模式吧。所以说，基本上每个 Linux distributions 除了架构的严谨度与选择的套件内容外，其实差异并不太大
